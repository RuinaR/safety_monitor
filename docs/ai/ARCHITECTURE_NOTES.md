# ARCHITECTURE_NOTES.md

## 현재 목표 구조

### Client

- 로컬 카메라 소유
- 객체 탐지 수행
- 서버로 소스 등록, 상태, 프리뷰, 프레임 탐지 결과 전송
- 현재 기본 정책: PC 1대당 클라이언트 1개, 클라이언트 1개당 카메라 0번 1개

### Server

- 소스 메타데이터 저장
- 소스 상태 저장
- 프레임 탐지 결과 저장
- 소스별 룰 적용 후 이벤트 판정 및 DB 저장
- 이벤트 클립 MP4 생성
- 이벤트 썸네일 JPG 생성
- 프리뷰 스트림, 이벤트, 상태, 클립, 썸네일 조회 API 제공
- 룰 설정 저장 API 제공

### Viewer

- 서버 데이터 기반 관제 UI
- 좌측 카메라 리스트 표시
- 중앙 Live Monitoring 영상 그리드 표시
- 우측 이벤트 로그/룰 설정 표시
- 이벤트 클릭 시 해당 카메라 타일에서 클립 재생
- 소스별 룰 설정 변경

## 현재 실제 분리 상태

### 서버

- FastAPI 앱: `safety_monitor_server/`
- 중앙 저장소:
  - `data/monitor.db`
  - `data/clips/`
  - `data/event_thumbnails/`
  - `data/source_previews/`
- 이벤트 판정 로직:
  - `app/server_event_processor.py`
- 이벤트 클립/썸네일 생성:
  - `app/server_clip_recorder.py`

### 클라이언트

- Flutter 앱: `safety_monitor_client/`
- 내장 Python 백엔드: `safety_monitor_client/embedded_backend/`
- 영상 원본 접근과 객체 탐지를 담당

### 뷰어

- Flutter 앱: `safety_monitor_viewer/`
- 서버 API와 서버 프리뷰 스트림만 사용
- 카메라 선택/이벤트 필터/클립 재생/룰 설정 담당

## 데이터 흐름

### 1. 소스 등록

- 클라이언트가 로컬 카메라 0번을 등록합니다.
- 서버가 `sources`에 저장합니다.
- `source_key`에는 클라이언트 소유자 정보가 포함됩니다.

### 2. 로컬 탐지

- 클라이언트가 카메라 프레임을 열고 객체 탐지를 수행합니다.
- 프리뷰 프레임을 서버로 보냅니다.
- 프레임 탐지 결과를 서버로 보냅니다.
- 상태 heartbeat를 서버로 보냅니다.

### 3. 서버 저장 및 이벤트 판정

- 서버가 소스 상태와 프레임 탐지 결과를 저장합니다.
- 서버가 `sources.payload_json.rule_config`를 기준으로 이벤트를 판정합니다.
- 이벤트를 `events`에 저장합니다.
- 이벤트가 종료되면 서버 프리뷰 버퍼에서 클립과 썸네일을 생성합니다.

### 4. 뷰어 조회/제어

- 뷰어는 서버 프리뷰 스트림을 표시합니다.
- 뷰어는 소스 상태와 이벤트 로그를 조회합니다.
- 뷰어는 이벤트 클릭 시 서버 클립을 재생합니다.
- 뷰어는 소스별 룰 설정을 서버에 저장합니다.

## 현재 설계 원칙

- 원본 영상 소유권은 클라이언트에 둡니다.
- 중앙 이벤트 판정과 저장은 서버에 둡니다.
- 뷰어는 객체 탐지를 수행하지 않습니다.
- 뷰어는 관제와 룰 설정 UI를 담당합니다.
- 룰 변경은 현재 시점 이후 이벤트 판정에 적용됩니다.
- 과거 이벤트 DB와 클립은 룰 변경 후에도 조회 가능합니다.
- 위험구역 이벤트는 이벤트 당시 ROI를 이벤트 payload에 저장해 과거 클립 오버레이가 현재 ROI 변경에 흔들리지 않게 합니다.

## 카메라 표시명 정책

현재 내부 식별자는 `source_key`입니다. UI에서는 `source_key`를 그대로 보여주지 않고 등록 순서 기반 `카메라 1`, `카메라 2`를 기본 표시명으로 사용합니다.

향후 권장 확장:

- `camera_slot_id`: 물리 위치/사용자 슬롯 식별자
- `display_name`: 사용자가 지정한 별칭
- `source_key`: 현재 연결된 클라이언트 소스 기술 키

이렇게 분리하면 PC가 교체되어도 기존 물리 카메라 슬롯에 새 클라이언트를 재매핑할 수 있습니다.

## 현재 한계

- DB는 SQLite 기반이라 대규모 동시 writer 환경에는 한계가 있습니다.
- 카메라 표시명은 아직 영구 별칭이 아니라 뷰어 표시 순번입니다.
- 클라이언트 headless 전환은 고려 중이지만 현재는 Flutter 클라이언트 GUI가 남아 있습니다.
